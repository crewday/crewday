# 18 — Internationalization

v1 ships English-only per the user's direction. This document describes
the **seam** that keeps i18n cheap to add later.

## Goals

- Make English-only feel first-class, not like a "missing feature".
- Ensure that adding Spanish, French, Portuguese, Tagalog, or Hindi
  later is a translation job, not a code refactor.

## Library choices

- **Babel** + `lingua` for extraction (`pybabel`).
- Jinja2 `i18n` extension with `{% trans %}` / `{{ _() }}`.
- `ICU MessageFormat` via `Babel` for plural/gender forms.
- BCP-47 locale tags (`en-US`, `es-ES`, `pt-BR`, `tl-PH`).

## Rules followed in v1 (even without other languages)

1. **All user-facing strings go through `_()`.** Tests assert no
   raw `<title>`, `<button>`, or text node missing the gettext call.
2. **Dates/numbers/currency via `Babel` formatters**, never via
   f-strings. Every timestamp display goes through a helper that takes
   locale + timezone; v1 defaults to `en-US` + property timezone.
3. **Plural forms via `ngettext`**, not `"%d items" if n else "1 item"`.
4. **Strings do not concatenate** to build sentences. Full sentences
   are single translation units with placeholders.
5. **Icon labels and alt text are translated**, not hard-coded.
6. **Model enum labels** live in a single `LOCALIZED_ENUMS` catalog
   referenced at render time, not duplicated into each template.
7. **Timezones** stored as IANA everywhere. Display in the property's
   timezone unless the user overrides.
8. **Locale resolution order**: per-user preference (future) →
   `Accept-Language` → household default → `en-US`.

## Per-employee language field

`employee.languages` is captured today (§05) as a BCP-47 array, used
for informational purposes in v1 (manager can see who speaks what).
It becomes the active UI locale source when translations land.

## Translation content for tasks and instructions

Deferred. Three possible designs are documented so we do not code
against one now:

- **Option A**: multi-lingual editor per task/instruction — one row
  per language on the `instruction_revision` table. Simplest. Costs
  manager time.
- **Option B**: keep one canonical language per task, translate on
  display with LLM; cache translations.
- **Option C**: both — managers can override per language for
  critical instructions, LLM auto-translates everything else.

v1 does not choose; the model in §07 leaves the door open by keeping
content on a separate `instruction_revision` row that could be
extended with a `locale` column.

## Right-to-left

- CSS uses logical properties (`margin-inline-start`) instead of
  `margin-left`.
- Icon directionality handled per icon (back arrow mirrors).
- Layout tested with `<html dir="rtl">` in the styleguide.

## Pseudolocale

Dev builds include a `qps-ploc` pseudo-locale that inflates English by
~30% and adds accents (`Ĥéllo Ŵörld!`) to stress layouts. Enable with
`?locale=qps-ploc`.

## Out of scope for v1

- Translator workflow (Crowdin, Weblate).
- Right-to-left testing beyond CSS.
- Voice / chat assistant multilingual support (capability exists but
  only English is committed).
